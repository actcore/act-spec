# ACT Spec v0.1.2 Design

## Decisions Made

### 1. Unified `metadata` type
All key-value extension points use a single type:
```wit
type metadata = list<tuple<string, list<u8>>>;
```
Field name is `metadata` everywhere. Well-known keys use `std:` prefix (e.g. `std:read-only`, `std:timeout-ms`). Third-party keys use their own namespace (e.g. `acme:priority`). Capability IDs use WASI-like naming (e.g. `wasi:sockets/tcp`).

Replaces: `extensions: list<tuple<string, string>>` (old pattern), `tool-annotations` record, response `headers`.

### 2. `error-kind` enum removed
`tool-error.kind` is now `string`. Well-known values (`std:not-found`, `std:invalid-args`, `std:timeout`, `std:capability-denied`, `std:internal`) documented in spec. Custom values use namespaced strings (e.g. `acme:rate-limited`).

### 3. `tool-annotations` record removed
`description` moves up to `tool-definition`. All other annotations become well-known keys in `tool-definition.metadata`:
- `std:read-only` (bool)
- `std:idempotent` (bool)
- `std:destructive` (bool)
- `std:usage-hints` (localized-string)
- `std:anti-usage-hints` (localized-string)
- `std:examples` (list of dCBOR)
- `std:tags` (list of strings)
- `std:timeout-ms` (u32)

### 4. `call-tool` returns `call-response` with unified stream errors
```wit
record call-response {
  metadata: metadata,
  body: stream<stream-event>,
}
call-tool: async func(config: option<list<u8>>, call: tool-call) -> call-response;
```
Response metadata always available. All errors (early and mid-stream) delivered as `stream-event::error` â€” no separate `result` wrapper. A `stream-event::error` is terminal.

### 5. `list-tools` returns `result` with response metadata, is async
```wit
record list-tools-response {
  metadata: metadata,
  tools: list<tool-definition>,
}
list-tools: async func(config: option<list<u8>>) -> result<list-tools-response, tool-error>;
```
Enables graceful error handling for bridge components. Async because bridge components may need to fetch remote schemas (OpenAPI, etc.) during discovery.

### 6. `get-config-schema` and `get-info` unchanged
Load-time metadata, synchronous, no need for response wrappers.

## Complete WIT

See `wit/act-core.wit` for the authoritative source. Key differences from the initial design:
- `component-metadata` interface merged into `tool-provider` (single interface, single world export)
- `component-capabilities` record removed; `capabilities: list<capability>` inlined into `component-info` with `required: bool` on each capability
- `parameter-meta` record removed; replaced by `parameters-schema: string` (JSON Schema)
- `call-response.body` simplified from `result<stream<stream-event>, tool-error>` to `stream<stream-event>` (unified stream error model)

- **Host-based tool chaining** â€” host provides an `act-consumer` WIT interface that components can import to call other components' tools directly, without going through the LLM. Enables efficient toolâ†’tool pipelines orchestrated by the host.
- **Resource provider** â€” separate `resource-provider` WIT interface for static/dynamic resources (icons, UI templates, files). First use case: component icon (`std:icon`). Future: MCP Apps-style UI resources.
- **OpenTelemetry integration** â€” host links [`wasi:otel`](https://github.com/WebAssembly/wasi-otel) imports (Phase 2) for components, enabling spans/metrics/logs from within Wasm. Protocol already supports W3C Trace Context via `std:traceparent`/`std:tracestate` metadata keys. Host propagates trace context across component boundaries and transport adapters.
- **Cost metadata** â€” complex topic: needs to support multiple currencies, crypto, token-based pricing, free tiers, rate limits. Too complex for `std:` keys â€” likely needs a dedicated billing/marketplace layer outside core protocol. Deferred indefinitely.
