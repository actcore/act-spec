# ACT Spec v0.1.3 Design

## Scope

Three additions to the ACT specification:

1. **Event Provider** — WIT interface for server→client push notifications
2. **Resource Provider** — WIT interface for static/dynamic resources
3. **Auth Best Practices** — standalone document for auth patterns

Multi-component HTTP endpoints removed from scope (deployment concern, not protocol).

## 1. Event Provider

### WIT Interface

```wit
interface event-provider {
  use types.{metadata, localized-string};

  record event-type-info {
    kind: string,
    description: localized-string,
    metadata: metadata,
  }

  record event {
    kind: string,
    data: option<list<u8>>,
    metadata: metadata,
  }

  get-event-types: func() -> list<event-type-info>;
  subscribe: async func(config: option<list<u8>>) -> stream<event>;
}
```

### Design Decisions

- **Fire-and-forget (push-only)** — no topic-based filtering, no subscription management. Client receives all events.
- **Stream-based** — component returns `stream<event>`, host reads. Native wasip3 stream, consistent with `call-tool`.
- **With config** — consistent with `list-tools` and `call-tool`. Bridge components need config for external subscriptions.
- **Minimal event record** — type + optional dCBOR payload + metadata. Timestamp and event ID via metadata conventions (`std:timestamp`, `std:event-id`) if needed.

### Well-Known Event Types

- `std:tools:changed` — tool list updated, clients should re-fetch

Custom event types use namespace prefix (e.g. `acme:order_updated`).

### HTTP Binding

`GET /events` → SSE stream:

```
event: std:tools:changed
data: {}

event: acme:order_updated
data: {"order_id": "123"}
```

### MCP Binding

Maps to MCP notifications (e.g. `notifications/tools/list_changed`).

## 2. Resource Provider

### WIT Interface

```wit
interface resource-provider {
  use types.{metadata, localized-string};

  record resource-info {
    uri: string,
    mime-type: option<string>,
    description: localized-string,
    metadata: metadata,
  }

  record resource-response {
    mime-type: option<string>,
    metadata: metadata,
    body: stream<u8>,
  }

  list-resources: async func(config: option<list<u8>>) -> list<resource-info>;
  get-resource: async func(config: option<list<u8>>, uri: string) -> resource-response;
}
```

### Design Decisions

- **URI-based** — generic, extensible. Well-known URIs for standard resources.
- **With config** — consistency with other async methods. Bridge components may load resources from external services.
- **Stream body** — resources can be large (UI templates, files). `stream<u8>` handles both small icons and large files.
- **Response record** — `resource-response` with mime-type + metadata + stream body. Analogous to `call-response`. Mime-type may differ from listing (content negotiation).

### Well-Known URIs

- `std:icon` — component icon (PNG/SVG)

### HTTP Binding

- `GET /resources` — list resources (returns `resource-info` array)
- `GET /resources/{uri}` — get resource (returns bytes with `Content-Type` header)

### MCP Binding

Maps to MCP resources.

## 3. Auth Best Practices (ACT-AUTH.md)

### Standalone Document

Referenced from ACT-SPEC, ACT-HTTP, ACT-TRANSPORT. Covers auth patterns across all transports.

### Two Levels of Auth

| | Auth to component | Auth to underlying service |
|---|---|---|
| Who authenticates | Client → host | Component → external API |
| Where | HTTP headers / transport | Config |
| Who validates | Host | External service |
| OIDC flow | Client performs flow, sends token to host | Host performs flow, passes token in config |

### Well-Known Config Schema Keys

For underlying service auth (config level):

- `std:api-key` — API key string
- `std:bearer-token` — OAuth/bearer token
- `std:username` + `std:password` — basic auth

### Document Sections

1. **Two levels of auth** — transport-level (client→host) vs config-level (component→external)
2. **Transport-level auth** — standard HTTP auth patterns, host policies, OIDC for component access
3. **Config-based auth patterns** — config-schema examples for each `std:` key pattern
4. **Host-level auth policies** — credential management (env vars, secret stores, per-component config), host-managed OIDC for underlying services
5. **Security considerations** — no secrets in metadata, config encryption at rest, HTTPS requirement, header vs body security trade-offs
6. **Examples** — full examples for API key, OAuth, basic auth, host-managed OIDC

### Out of Scope

OAuth/OIDC flow details — reference RFC 6749 / OpenID Connect Core. ACT receives ready tokens.

## Updated World

```wit
world act-world {
  export tool-provider;
  export event-provider;      // optional
  export resource-provider;   // optional
}
```

`event-provider` and `resource-provider` are optional exports. A minimal component only exports `tool-provider`.
